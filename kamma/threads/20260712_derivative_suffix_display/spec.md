# Spec: Derivative (suffix) grammar row displays correctly

## Overview
In the grammar table, the "Derivative" row renders incorrectly. The webapp shows
`{derivative} ({suffix})` — e.g. `kita (ṇvu)` — but the Flutter app shows only the
suffix in parentheses — e.g. `(ṇvu)` — with the derivative type missing entirely.

There are two independent causes, one in each repo:

1. **Data is stripped from the mobile DB.** The mobile exporter explicitly drops the
   `derivative` column (`../dpd-db/exporter/mobile/mobile_exporter.py:27`,
   `HEADWORD_COLUMNS`). `suffix` is kept, `derivative` never reaches the phone.
   Consequently the app's Drift schema (`tables.dart`, `database.g.dart`) has no
   `derivative` column at all — the value does not exist on-device.

2. **The UI was coded around the missing column.** `_buildDerivativeRow`
   (`lib/widgets/grammar_table.dart:186-193`) gates on `suffix` (the webapp gates on
   `derivative`) and outputs only `($suffix)`, so even a valid suffix shows as a lone
   parenthesised affix with no derivative label.

Fix both so the row matches the webapp.

## Repo context
- Webapp reference — `../dpd-db/exporter/webapp/templates/dpd_headword.html:1094-1097`:
  ```html
  {% if d.i.derivative %}
      <td>{{d.i.derivative}} ({{d.i.suffix}})</td>
  ```
  Row renders only when `derivative` is non-empty.
- Source column exists in the full DB: `../dpd-db/db/models.py:1043` (`derivative`),
  `:1044` (`suffix`).
- Mobile exporter keeps `suffix` (`mobile_exporter.py:53`) but the `Dropped:` comment
  (`:27`) lists `derivative`; it is absent from `HEADWORD_COLUMNS`.
- Schema versions are coupled and MUST stay in sync:
  - `mobile_exporter.py:129` — `DB_SCHEMA_VERSION: int = 6`
  - `lib/database/database.dart:35` — `requiredDbSchemaVersion = 6`; `:42` — `schemaVersion => 6`
- App schema/getters: `lib/database/tables.dart:45` (`suffix` column, add `derivative`
  next to it); `lib/database/dao.dart:808` (`suffix` getter, add `derivative`);
  `lib/database/database.g.dart` is Drift-generated — regenerated by `build_runner`.
- DPD DB uses empty strings, never NULL — the webapp `{% if %}` and the app's
  `isEmpty` gate both treat `''` as absent, which is correct here.

## What it should do
1. The mobile DB export includes the `derivative` column for every headword.
2. The app schema exposes `derivative`, and `_buildDerivativeRow` renders
   `$derivative ($suffix)`, gated on `derivative` being non-empty (same row-visibility
   rule as the webapp). One deliberate difference from a literal port: when `derivative`
   is present but `suffix` is empty (~871 rows), show just `$derivative` with no
   parentheses — the webapp template has no guard and would render a dangling `()`.
3. Niggahita/apostrophe filtering (`filter: n`) still applies to the rendered text.

## Assumptions & uncertainties
- Adding one column to the mobile export requires bumping the DB schema version
  (`6 → 7`) in both `mobile_exporter.py` and `database.dart`, so the app pulls a fresh
  DB rather than reusing the incompatible v6 one already on-device.
- The full desktop `dpd.db` already has `derivative` populated; only the mobile export
  needs to stop dropping it. No upstream data work required — flag at review if any
  headword unexpectedly has an empty `derivative` but a non-empty `suffix`.
- The label stays "Derivative" (already matches the webapp row label). No wording change.

## Constraints
- Native Flutter widget only — no HTML. Reuses the existing `buildKvTextRow` path.
- No behaviour change to any other grammar row.
- `pubspec.lock` untouched; `database.g.dart` regenerated via build_runner, never hand-edited.

## How we'll know it's done
- After rebuilding the mobile DB and reinstalling, a word with a derivative (e.g. a
  kita/taddhita form) shows `Derivative  kita (ṇvu)` in the grammar table, matching the webapp.
- Words with no derivative show no Derivative row (unchanged).
- `flutter analyze` clean; `flutter test` passes; `build_runner` regenerates cleanly.
